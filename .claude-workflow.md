## 2. Claude's Role

Claude acts as the **backend brain** for the frontend. There are no APIs, no database, no server logic yet. Claude:
- Writes all code changes (HTML, CSS, JS)
- Manages localStorage data via migrations
- Triages feedback into Backlog
- Updates milestones, changelog, tester
- Provides commit descriptions for the user to push via Visual Studio

**Future state** will replace Claude with proper APIs, database (Cloudflare D1 or similar), and server infrastructure. The current prototype uses localStorage for all state.

## 4. Dev/Test Workflow

### GO / COMMIT Signals (v0.78.0 — non-negotiable)

Claude does NOT start coding until Christian says **GO** (or "do it", "execute", etc.).
Claude does NOT package a commit message until Christian says **COMMIT** (or "commit that", etc.).

**Why this matters:** Claude's work launches a big process (file edits, tool calls, rendering). Starting too early while Christian is mid-thought wastes time and forces him to wait. Discussion and planning are cheap — executing prematurely is expensive.

**The rule:**
- **Before GO** → Claude discusses, asks questions, takes notes, plans. Hands off the code.
- **GO** → Claude starts coding. Execute the agreed plan.
- **COMMIT** → Claude packages the commit message for deploy.

If Christian is giving feedback or making notes without saying GO, Claude should ask: *"Want me to GO on this, or are you still thinking?"*

### The Flow
```
0. Claude reads Tester feedback FIRST (user can add feedback at any time — pick it up before starting work)
0.5 Discussion → Christian says GO → Claude begins work
1. Claude makes code changes
2. Claude updates: version (all 8 pages), changelog, tester sections, migrations
3. Christian says COMMIT → Claude provides commit description in a code block
4. Claude shows DEPLOY POPUP → User commits, pushes, deploys → clicks "Deployed"
5. Claude self-tests on live site (see Self-Testing below)
6. If FAILs exist → REWORK LOOP (see below)
7. Once all items PASS → User clicks PASS/FAIL buttons + "Complete Testing"
8. Claude shows TESTING STATUS POPUP → User clicks "Testing Complete" (or "Back to you" / "Hold on")
9. Claude processes: triages feedback → Backlog, clears feedback via migration
10. Claude delivers LESSONS LEARNED summary for the completed version (see Lessons Learned below)
11. Claude reviews PQ: move completed items to Recently Completed, update statuses, assess remaining work
12. Repeat from step 0
```

### Self-Testing Protocol (v0.64.7, upgraded v0.64.8, formalized v0.68.0)
Claude does the verification work, user does the clicking:
1. Claude navigates to the live site in Chrome after user pushes
2. Claude verifies each test item (inspect pages, check localStorage, read values)
3. Claude enters notes into each test item's note field on the Tester page
4. Claude **refreshes the Tester page** after entering notes (so user sees updates)
5. Claude reports PASS/FAIL recommendations with reasoning to the user in chat
6. **User clicks** the PASS/FAIL/SKIP buttons (Claude does NOT click these — user events fire properly)
7. User clicks "Complete Testing" when satisfied

### Rework Loop (formalized v0.68.0)
When Claude's self-test finds FAILs, rework happens WITHIN the same version:
```
1. Claude reports FAILs with details
2. Claude fixes the code (same version — no version bump)
3. Claude provides updated commit description
4. User commits, pushes, deploys
5. Claude re-tests ONLY the previously-failed items
6. Claude enters updated notes, reports results
7. If still failing → repeat from step 2
8. Once all PASS → user clicks buttons + Complete Testing
```
Key rules:
- Rework stays within the current version (no version increment)
- Only re-test items that were FAIL — don't re-test items already PASS
- HALT banner stays RED throughout the rework cycle
- User clicks Complete Testing only when all items are resolved (PASS or SKIP)

### Lessons Learned Summary (T-090, formalized v0.80.0)
After testing is complete and feedback is triaged (step 9), Claude delivers a brief Lessons Learned summary before reviewing the PQ (step 11). This is a reflection moment — celebrate wins, flag patterns, and capture improvements while the version is fresh.

**Format:**
```
vX.XX.X — Lessons Learned

What was delivered: (1-2 lines — what shipped and why it matters)
What went well: (smooth areas, good patterns, efficient execution)
What could improve: (friction, rework causes, missed steps, wasted effort)
Patterns/gotchas: (anything worth adding to Section 10 Key Patterns & Gotchas)
Celebration: (acknowledge effort, progress toward milestones, momentum)
```

**Rules:**
- Keep it concise — 5-10 bullets max across all sections
- Skip any section that has nothing meaningful to say (don't pad)
- If a gotcha is broadly applicable, add it to Section 10 in the same edit
- If an improvement suggests a process change, note it but don't act — Christian decides
- This is NOT a report — it's a quick retro. Don't over-produce it.

### Bookend Scans (formalized v0.69.0)
The user is live — adding feedback, updating PQ, and adjusting backlog items while Claude is heads-down coding. A single check at version start isn't enough. Claude scans **before AND after** every work cycle.

**PRE-WORK SCAN** — Before starting any version:
- Read PQ for current/new tasks
- Read Feedback for new items
- Scan Backlog for changes or new items added by user
- Triage anything pending (feedback → backlog, clear resolved items)

**POST-WORK SCAN** — After testing completes, before showing Testing Status popup:
- Re-check Feedback for anything added during the work cycle
- Re-check PQ for new/changed items
- Triage new feedback → backlog if needed
- Clear resolved feedback items
- Include quick additions in current version if they fit, otherwise note for next version

### Handoff Popups (formalized v0.68.0)
Claude uses AskUserQuestion popups at key handoff points to reduce typing:

**DEPLOY POPUP** — shown after commit description, waits for user to deploy:
- "Deployed" → Claude begins self-testing
- "Hold on" → Claude waits

**TESTING STATUS POPUP** — shown after Claude reports test results (or after rework re-test):
- "Testing Complete" → Claude proceeds to process feedback, plan next version
- "Back to you" → User has an issue, feedback, or needs something adjusted first
- "Hold on" → User needs more time to review

**REWORK DEPLOY POPUP** — shown during rework loop after fix commit description:
- "Deployed" → Claude re-tests only the previously-failed items
- "Hold on" → Claude waits

### Task List Visibility
Claude keeps the TodoWrite task list active and visible throughout the session. The task list appears in the upper-right of Claude Desktop and gives the user real-time visibility into what Claude is doing. Key rules:
- Always have at least one todo item visible during active work
- Don't let the list go empty until the session is truly idle
- Use "Waiting for..." items when the ball is in the user's court

### HALT Banner Protocol
- **RED (pulsing)** = Testing in progress — DO NOT modify code
- **GREEN** = Testing complete — ready for dev
- **GREY** = Idle / waiting for activity

### Self-Test Report Format (locked v0.70.0)
Claude reports self-test results using this exact format:

**Title line**: `vX.XX.X Self-Test Results — All PASS:` (or `X FAIL` if applicable)

**Markdown table** with 3 columns: `#`, `Test`, `Result`
- Each row = one test item
- Result = ✅ PASS or ❌ FAIL (with brief note if needed)

**Summary line**: `Testing Status: COMPLETE — X/Y PASS. Tester is loaded and ready for your click-through when you're ready.`

### Commit Description Format
Always provide in a code block so user can copy/paste to Visual Studio:
```
v0.XX.X — Short Title

TYPE: Description of change
TYPE: Description of change
...
```
Types: FEATURE, FIX, UPDATE, RENAME, INFRA

## 5. Data Architecture

### localStorage Keys
| Key | Purpose |
|-----|---------|
| `opsflow_tasks_v2` | Tasks page data (PQ + Backlog items) |
| `opsflow_tasks_migrations` | Tracks which migrations have run (array of keys) |
| `opsflow_feedback` | Flat array of feedback items from tester |
| `opsflow_tester` | Test results per version |
| `opsflow_tester_halt` | HALT banner state (testing/complete/idle) |
| `opsflow_tester_fingerprint` | Auto-reset fingerprint for test sections |
| `opsflow_tester_config` | Optional override for TEST_VERSION & TEST_SECTIONS |
| `opsflow_upload_secret` | Shared secret for Worker API auth (set once per browser) |
| `opsflow_version` | Suite version string (synced to/from KV) |
| `opsflow_launchpad` | LaunchPad sphere config (tools + personal sections) |
| `opsflow_workshop` | Workshop card data |

### Cloudflare KV Backup (v0.77.0 — T-085/T-086)
- **KV is the cloud backup** for localStorage data
- `shared/opsflow-sync.js` loaded on every page via `OpsFlow.init()`
- Auto-backs up localStorage to KV on page unload (via sendBeacon)
- Auto-restores from KV if localStorage is empty (recovery mode)
- Version number stored in KV key `opsflow_version`, read dynamically by all pages
- Secret stored in browser: `localStorage.setItem('opsflow_upload_secret', 'THE_SECRET')`
- Worker API: `api.opsflowbetter.com/kv/:key` (GET/PUT/DELETE), `/kv-bulk` (POST), `/kv-list` (GET)
- All endpoints require `X-Upload-Secret` header

### HTML as Safety Net (Critical Pattern)
- Static HTML is the **fallback source of truth**
- If localStorage is lost, pages re-seed from HTML on load
- **ALWAYS keep both in sync**: update HTML defaults AND create localStorage migration
- This dual-write pattern ensures resilience
- **Future direction**: KV replaces HTML-as-backup (T-086). HTML defaults remain as last resort.

### Migration System
Located in `tasks.html` inside `runMigrations()`:
- Each migration has a unique key (e.g., `v0.64.7_inject_feedback_items`)
- Runs once per browser via `opsflow_tasks_migrations` tracking
- Idempotent — safe to re-run, won't duplicate items
- Used to: inject new backlog items, clear processed feedback, modify existing data

### Tester Auto-Reset Fingerprint
- Generates a fingerprint from `TEST_SECTIONS` content
- If sections change (new test items, different wording), auto-clears stale test results
- Preserves feedback — only clears test data for the current version

## 6. Feedback Pipeline

### User submits feedback on Tester page:
- Category: Bug / Improvement / Todo / Question / Note
- Priority: Now (red) / Soon (orange) / Later (blue)

### Claude processes feedback:
1. Read feedback from live site
2. Create Backlog items with matching priority:
   - Now → `priority: 'high'`
   - Soon → `priority: 'medium'`
   - Later → `priority: 'low'`
3. Add to HTML defaults (in correct priority section with comments)
4. Create migration to inject into localStorage
5. Create migration to clear the feedback items from `opsflow_feedback`
6. User promotes from Backlog → Up Next when ready (their call, not Claude's)

### Priority LED Labels
- Backlog and PQ items show clickable priority LEDs with Now/Soon/Later text
- Click cycles: Now → Soon → Later → Now
- Backlog uses **2-second debounced re-sort** (items stay in place while cycling, then re-sort after 2s of no clicks)
- PQ does NOT sort by priority (user controls order via dropdown)

## 10. Key Patterns & Gotchas

- **Claude NEVER commits or pushes from the sandbox** — Claude provides the commit message in a code block. Christian commits and pushes from Visual Studio. This is non-negotiable. The sandbox creates lock files and LF line endings that break VS. Established as a work rule in v0.77.0.
- **CRLF line endings** — If VS ever shows phantom "changed" files, run `git config core.autocrlf true` in the repo, close VS, delete `.git/index.lock` and `.git/HEAD.lock`, then `git checkout -- .` to normalize.
- **Never use `-uall` flag** with git status (memory issues on large repos)
- **Read files before editing** — Edit tool requires prior Read
- **sed for simple replacements**, Read+Edit for complex changes
- **HTML entities in tasks.html** — use `&amp;`, `&rsquo;`, `&lsquo;`, `&mdash;`, etc.
- **Backlog HTML is ordered**: `<!-- NOW (high) -->`, `<!-- SOON (medium) -->`, `<!-- LATER (low) -->` section comments
- **Priority sort values must be non-zero** — `{ high: 1, medium: 2, low: 3 }` not `{ high: 0, ... }` because `0 || default` is falsy in JS
- **Don't clear feedback you misunderstood** — if the Backlog item description doesn't match user intent, the feedback clearing was premature
- **Self-test before handoff** — Claude should verify on live site: version lines, sort behavior, new backlog items, LED clicks
- **Read Tester results before claiming PASS** — Christian can override Claude's test results. Always re-read the Tester page after self-test before declaring final results.
- **"Rework" not "fix"** — post-deploy corrections are called reworks, not new features or fixes
- **Check time of day** when greeting Christian — don't say "good evening" at 9am
- **Don't burn credits on chat** — stay focused, avoid unnecessary back-and-forth
- **Migration keys must be unique** and should include version number
- **Migrations must be tested against real data** — read actual localStorage values before writing indexOf matches. Never assume field content.
- **Feedback clearing migrations** use `indexOf()` on `fb[i].text` to match
- **Tester fingerprint auto-resets** when TEST_SECTIONS content changes — no manual reset needed
- **PQ item IDs** start with `pq_`, **Backlog IDs** start with `bl_`, **Completed IDs** start with `done_seed_` (HTML-seeded) or keep original ID (migration-completed)
- **Migration-injected items** should use descriptive stable IDs (e.g., `backlog-delete-action`)
- **Tester completion stamp** — Claude calls `completeAITest()` after self-testing (purple "AI Self-Test" banner). Christian calls `completeTesting()` for manual tests (green "Manual Test" banner). History shows [AI] or [Manual] tags.
