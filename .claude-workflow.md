## 2. Claude's Role

Claude acts as the **backend brain** for the frontend. There are no APIs, no database, no server logic yet. Claude:
- Writes all code changes (HTML, CSS, JS)
- Manages localStorage data via migrations
- Triages feedback into Backlog
- Updates milestones, changelog, tester
- Provides commit descriptions for the user to push via Visual Studio

**Future state** will replace Claude with proper APIs, database (Cloudflare D1 or similar), and server infrastructure. The current prototype uses localStorage for all state.

## 4. Dev/Test Workflow

### GO / COMMIT Signals (v0.78.0 — non-negotiable)

Claude does NOT start coding until Christian says **GO** (or "do it", "execute", etc.).
Claude does NOT package a commit message until Christian says **COMMIT** (or "commit that", etc.).

**Why this matters:** Claude's work launches a big process (file edits, tool calls, rendering). Starting too early while Christian is mid-thought wastes time and forces him to wait. Discussion and planning are cheap — executing prematurely is expensive.

**The rule:**
- **Before GO** → Claude discusses, asks questions, takes notes, plans. Hands off the code.
- **GO** → Claude starts coding. Execute the agreed plan.
- **COMMIT** → Claude packages the commit message for deploy.

If Christian is giving feedback or making notes without saying GO, Claude should ask: *"Want me to GO on this, or are you still thinking?"*

### The Flow
```
0. Claude reads WFD FIRST — check action queue, open discussions, story state. Then read Tester feedback.
0.5 Discussion → Christian says GO → Claude builds plan WITH embedded workflow actions (status updates, version bumps, changelog, tester config, migrations, DOC UPDATES) as explicit steps in the plan — then executes
1. Claude makes code changes (first action: set PQ item to In Progress via migration)
2. Claude updates: version (all pages), changelog, tester sections, migrations
3. Christian says COMMIT → Claude queues commit via OpsFlow action queue on WFD (see Deploy Handoff Message Format)
3.5 Claude asks: "Checkpoint deploy or tasks complete?" (see Checkpoint vs Completion below)
4. Claude shows DEPLOY POPUP → User commits, pushes, deploys → clicks "Done ✓" on WFD action banner
5a. If CHECKPOINT → Claude runs OpsFlow.setVersion, verifies deploy works, tasks stay at current status (Deployed or In Progress). Resume work from step 1.
5b. If TASKS COMPLETE → Claude runs OpsFlow.setVersion('X.XX.X') on live site, then self-tests (see Self-Testing below)
** EVERY COMMIT: Claude updates .claude-workflow.md AND APPLICATION-DESIGN.md if anything changed about the process, architecture, or WFD. This is NOT optional. Stale docs = broken recovery. **
6. If FAILs exist → REWORK LOOP (see below)
7. Once all items PASS → User clicks PASS/FAIL buttons + "Complete Testing"
8. Claude shows TESTING STATUS POPUP → User clicks "Testing Complete" (or "Back to you" / "Hold on")
8.5 If "Back to you" → Claude RE-READS TESTER for FAILs. If FAILs → REWORK LOOP. If user feedback → address it.
9. Claude processes: triages feedback → Backlog, clears feedback via migration
10. Claude delivers LESSONS LEARNED summary for the completed version (see Lessons Learned below)
11. Claude reviews PQ: set completed items to Done. Christian clicks Archive when ready (Claude NEVER archives).
12. Repeat from step 0
```

### Task Status Lifecycle (T-091, formalized v0.81.0, expanded v0.89.0)
PQ and Backlog items have a clickable status badge that cycles through states. Both Claude and Christian update statuses — the PQ is a live communication channel. No chat needed for status updates.

**Status cycle**: New → Discussing → In Progress → Deployed → Testing → Rework → *(loops back to In Progress)* → Done *(terminal)*
**Archive**: Items at "Done" show an Archive button. User clicks Archive to move the item to Recently Completed.
**Valid code values**: `new`, `discussing`, `in-progress`, `deployed`, `testing`, `rework`, `done`

**Who sets what:**
- **Discussing** → Claude (when discussion starts on a task)
- **In Progress** → Claude (when coding starts — before any code is written)
- **Deployed** → Christian (clicks badge after deploy — signals Claude to proceed)
- **Testing** → Claude (when self-testing begins on the group)
- **Rework** → Claude (when testing finds failures)
- **Done** → Claude (ONLY after Christian confirms Testing Complete — never before)

**Rework loop**: Testing → Rework → In Progress (fix) → Deployed (Christian) → Testing (re-test) → Done (all pass)

**Rules:**
- Claude updates status via live JS on the site or via migrations in tasks.html
- "Done" is terminal — clicking the status badge at "Done" does nothing. Only Archive moves it out.
- Christian or Claude can always cycle status by clicking the badge (except at Done)
- Claude NEVER sets Done until Testing Complete is confirmed — deployed but untested stays In Progress or Deployed

### Group Batch Workflow (v0.88.1)
When the PQ is loaded by group (e.g., all Admin tasks), work and deploy tasks individually but batch the testing/changelog/tester workflow to the end of the group.

**The pattern:**
1. Promote a group of related tasks to PQ (e.g., Admin group: T-100, T-101, T-102, T-103)
2. Set parent story to **Discussing**, child tasks stay **New**
3. When work starts on a child task → Claude sets it to **In Progress**
4. Code, commit — Claude provides commit message (format: `vX.XX.X T-### description`)
5. Christian deploys → Christian clicks badge to **Deployed** (signals Claude)
6. Claude reads PQ, sees Deployed → moves to next task or starts batch workflow
7. After ALL tasks in the group are Deployed → Claude batches the workflow:
   - Update changelog (one entry covering the group, or per-deploy entries)
   - Set tester config covering all tasks in the group
   - Update workflow doc
8. Claude sets all group tasks to **Testing**, runs self-test against the full group
9. Rework loop if needed — Claude sets failing tasks to **Rework**, fixes, new commit
10. Christian marks **Testing Complete** → Claude sets all group tasks to **Done**
11. Archive all group tasks

**Why:** Testing the group as a whole is more meaningful than testing fragments. The PQ badges replace chat messages for status communication — Christian clicks Deployed instead of typing "DEPLOY COMPLETE".

### Self-Testing Protocol (v0.64.7, upgraded v0.64.8, formalized v0.68.0)
Claude does the verification work, user does the clicking:
1. Claude navigates to the live site in Chrome after user pushes
1.5 Claude runs `OpsFlow.setVersion('X.XX.X')` on any page to update KV + localStorage with new version (prevents stale version display)
2. Claude verifies each test item (inspect pages, check localStorage, read values)
3. Claude enters notes into each test item's note field on the Tester page
4. Claude **refreshes the Tester page** after entering notes (so user sees updates)
5. Claude reports PASS/FAIL recommendations with reasoning to the user in chat
6. **User clicks** the PASS/FAIL/SKIP buttons (Claude does NOT click these — user events fire properly)
7. User clicks "Complete Testing" when satisfied

### Rework Loop (formalized v0.68.0)
When Claude's self-test finds FAILs, rework happens WITHIN the same version:
```
1. Claude reports FAILs with details
2. Claude fixes the code (same version — no version bump)
3. Claude provides updated commit description
4. User commits, pushes, deploys
5. Claude re-tests ONLY the previously-failed items
6. Claude enters updated notes, reports results
7. If still failing → repeat from step 2
8. Once all PASS → user clicks buttons + Complete Testing
```
Key rules:
- Rework stays within the current version (no version increment)
- Only re-test items that were FAIL — don't re-test items already PASS
- HALT banner stays RED throughout the rework cycle
- User clicks Complete Testing only when all items are resolved (PASS or SKIP)

### Lessons Learned Summary (T-090, formalized v0.80.0)
After testing is complete and feedback is triaged (step 9), Claude delivers a brief Lessons Learned summary before reviewing the PQ (step 11). This is a reflection moment — celebrate wins, flag patterns, and capture improvements while the version is fresh.

**Format:**
```
vX.XX.X — Lessons Learned

What was delivered: (1-2 lines — what shipped and why it matters)
What went well: (smooth areas, good patterns, efficient execution)
What could improve: (friction, rework causes, missed steps, wasted effort)
Patterns/gotchas: (anything worth adding to Section 10 Key Patterns & Gotchas)
Celebration: (acknowledge effort, progress toward milestones, momentum)
```

**Rules:**
- Keep it concise — 5-10 bullets max across all sections
- Skip any section that has nothing meaningful to say (don't pad)
- If a gotcha is broadly applicable, add it to Section 10 in the same edit
- If an improvement suggests a process change, note it but don't act — Christian decides
- This is NOT a report — it's a quick retro. Don't over-produce it.

### AI Assist Prompt R&D (T-095, formalized v0.85.0)
Before coding ANY AI Assist feature (prompt-driven automation, LLM integration, AI-powered pipeline steps), the prompt MUST be tested by hand first. This prevents expensive rework cycles from untested prompt assumptions.

**Steps:**
1. Write the draft prompt (what Claude/GPT/Grok will receive)
2. Test it manually in at least 2 LLMs (e.g., Claude + ChatGPT or Grok)
3. Evaluate output: format, accuracy, length, consistency across models
4. Iterate the prompt until output is stable and matches expectations
5. Lock the prompt format — document what works and what doesn't
6. THEN code the integration (API call, UI, parsing logic)

**Rules:**
- No coding until the prompt is locked. Discussion and prompt drafts are free — rework after coding is expensive.
- Document prompt iterations briefly (what changed, why) so future sessions don't re-discover the same dead ends.
- If a prompt works in one LLM but not another, note the divergence — it affects which model we target.
- This applies to new AI Assist features AND fixes/rework of existing prompts.

### Bookend Scans (formalized v0.69.0)
The user is live — adding feedback, updating PQ, and adjusting backlog items while Claude is heads-down coding. A single check at version start isn't enough. Claude scans **before AND after** every work cycle.

**PRE-WORK SCAN** — Before starting any version:
- Read PQ for current/new tasks
- Read Feedback for new items
- Scan Backlog for changes or new items added by user
- Triage anything pending (feedback → backlog, clear resolved items)

**POST-WORK SCAN** — After testing completes, before showing Testing Status popup:
- Re-check Feedback for anything added during the work cycle
- Re-check PQ for new/changed items
- Triage new feedback → backlog if needed
- Clear resolved feedback items
- Include quick additions in current version if they fit, otherwise note for next version

### Handoff Popups (formalized v0.68.0)
Claude uses AskUserQuestion popups at key handoff points to reduce typing:

**DEPLOY POPUP** — shown after commit description, waits for user to deploy:
- "Deployed" → Claude begins self-testing
- "Hold on" → Claude waits

**TESTING STATUS POPUP** — shown after Claude reports test results (or after rework re-test):
- "Testing Complete" → Claude proceeds to process feedback, plan next version
- "Back to you" → User has an issue, feedback, or needs something adjusted first
- "Hold on" → User needs more time to review

**REWORK DEPLOY POPUP** — shown during rework loop after fix commit description:
- "Deployed" → Claude re-tests only the previously-failed items
- "Hold on" → Claude waits

### Task List Visibility
Claude keeps the TodoWrite task list active and visible throughout the session. The task list appears in the upper-right of Claude Desktop and gives the user real-time visibility into what Claude is doing. Key rules:
- Always have at least one todo item visible during active work
- Don't let the list go empty until the session is truly idle
- Use "Waiting for..." items when the ball is in the user's court

### HALT Banner Protocol
- **RED (pulsing)** = Testing in progress — DO NOT modify code
- **GREEN** = Testing complete — ready for dev
- **GREY** = Idle / waiting for activity

### Self-Test Report Format (locked v0.70.0)
Claude reports self-test results using this exact format:

**Title line**: `vX.XX.X Self-Test Results — All PASS:` (or `X FAIL` if applicable)

**Markdown table** with 3 columns: `#`, `Test`, `Result`
- Each row = one test item
- Result = ✅ PASS or ❌ FAIL (with brief note if needed)

**Summary line**: `Testing Status: COMPLETE — X/Y PASS. Tester is loaded and ready for your click-through when you're ready.`

### Commit Description Format
**See Deploy Handoff Message Format above** for the current process (action queue on WFD).

Commit message format (used in `action.detail` and as legacy code-block fallback):
```
vX.XX.X T-### Short description

TYPE: Description of change
TYPE: Description of change
```
Types: FEATURE, FIX, UPDATE, RENAME, INFRA

## 5. Data Architecture

### localStorage Keys
| Key | Purpose |
|-----|---------|
| `opsflow_tasks_v2` | Tasks page data (PQ + Backlog + Completed items) |
| `opsflow_tasks_migrations` | Tracks which migrations have run (array of keys) |
| `opsflow_feedback` | Flat array of feedback items from tester |
| `opsflow_tester` | Test results per version |
| `opsflow_tester_halt` | HALT banner state (testing/complete/idle) |
| `opsflow_tester_fingerprint` | Auto-reset fingerprint for test sections |
| `opsflow_tester_config` | Optional override for TEST_VERSION & TEST_SECTIONS |
| `opsflow_upload_secret` | Shared secret for Worker API auth (set once per browser) |
| `opsflow_version` | Suite version string (synced to/from KV) |
| `opsflow_launchpad` | LaunchPad sphere config (tools + personal sections) |
| `workshopState_v2` | Workshop card data |
| `opsflow_changelog` | Changelog entries (seeded from shared/changelog-data.json) |
| `opsflow_action_queue` | WFD action queue — single object `{id, type, message, detail, status, created}` |
| `opsflow_workflow_state` | WFD story state — trunk nodes, branches (checkpoints + discussions), version history |

### KV Cache Architecture (v0.82.0 — T-089)
- **KV is the source of truth** → localStorage is cache → pages render from cache
- `shared/opsflow-sync.js` loaded BEFORE main script on every page
- `OpsFlow.get(key)` — async, checks cache freshness, falls back to KV if stale (5 min threshold)
- `OpsFlow.getLocal(key)` — sync, localStorage only (for hot paths)
- `OpsFlow.set(key, data)` — writes to BOTH localStorage + KV
- Auto-backs up all SYNC_KEYS to KV on page unload (via sendBeacon)
- Auto-restores from KV if localStorage is empty (recovery mode)
- Version number: `OpsFlow.setVersion()` / `OpsFlow.getVersion()`
- Secret stored in browser: `localStorage.setItem('opsflow_upload_secret', 'THE_SECRET')`
- Worker API: `api.opsflowbetter.com/kv/:key` (GET/PUT/DELETE), `/kv-bulk` (POST), `/kv-list` (GET)

### Data-Driven Pages (v0.82.0)
- **Changelog**: Rendered from `opsflow_changelog` (localStorage/KV), seeded from `shared/changelog-data.json`
- **Tester**: TEST_VERSION + TEST_SECTIONS from `opsflow_tester_config`, seeded by migration
- **All pages**: Version display via `<span class="suite-ver">` filled by `OpsFlow.init()`
- **HTML contains zero data** — pages are pure templates. VS Code diffs show only real code changes.
- **Future direction**: KV swapped for D1 database. OpsFlow API stays the same.

### Migration System
Located in `tasks.html` inside `runMigrations()`:
- Each migration has a unique key (e.g., `v0.64.7_inject_feedback_items`)
- Runs once per browser via `opsflow_tasks_migrations` tracking
- Idempotent — safe to re-run, won't duplicate items
- Used to: inject new backlog items, clear processed feedback, modify existing data

### Tester Auto-Reset Fingerprint
- Generates a fingerprint from `TEST_SECTIONS` content
- If sections change (new test items, different wording), auto-clears stale test results
- Preserves feedback — only clears test data for the current version

## 6. Feedback Pipeline

### User submits feedback on Tester page:
- Category: Bug / Improvement / Todo / Question / Note
- Priority: Now (red) / Soon (orange) / Later (blue)

### Claude processes feedback:
1. Read feedback from live site
2. Create Backlog items with matching priority:
   - Now → `priority: 'high'`
   - Soon → `priority: 'medium'`
   - Later → `priority: 'low'`
3. Add to HTML defaults (in correct priority section with comments)
4. Create migration to inject into localStorage
5. Create migration to clear the feedback items from `opsflow_feedback`
6. User promotes from Backlog → Up Next when ready (their call, not Claude's)

### Priority LED Labels
- Backlog and PQ items show clickable priority LEDs with Now/Soon/Later text
- Click cycles: Now → Soon → Later → Now
- Backlog uses **2-second debounced re-sort** (items stay in place while cycling, then re-sort after 2s of no clicks)
- PQ does NOT sort by priority (user controls order via dropdown)

## 7. Version Bumping Checklist (v0.82.0 — data-driven)

Version bumps are now data-driven. Pages are pure templates — no HTML-as-data edits.

When making code changes:
1. Increment version number (e.g., v0.81.0 → v0.82.0)
2. **NO manual version edits in HTML** — `OpsFlow.setVersion()` updates KV + localStorage, `OpsFlow.init()` fills all `<span class="suite-ver">` elements dynamically
3. Update `shared/changelog-data.json` — prepend new entry with version, date, tag, title, items
4. Create migration in `tasks.html` to seed `opsflow_tester_config` with TEST_VERSION and TEST_SECTIONS. **Always include a "User Test" section** (see below)
5. Create any other needed migrations in `tasks.html` (status updates, backlog completions, etc.)
6. After deploy: run `OpsFlow.setVersion('X.XX.X')` on live site to update KV

**User Test Item (T-093, formalized v0.85.0):**
Every tester config MUST include a "User Test" section as the LAST section:
```javascript
{ title: 'User Test', priority: 'MANUAL', items: ['User verification \u2014 annotate with findings'] }
```
This gives Christian a dedicated slot to PASS/FAIL and annotate his own findings during testing. It counts toward the Complete Testing button's total (all items must be resolved). Claude should never pre-click this item — it's exclusively for the user.

**Files changed per version bump** (target: 2-3 files max):
- `shared/changelog-data.json` — new changelog entry
- `tasks.html` — migrations (tester config, status updates, backlog moves)
- Feature file(s) — the actual code changes

**Files that should NOT change for a version bump:**
- changelog.html (renders from data)
- tester.html (reads config from localStorage/KV)
- All other pages (version filled dynamically by opsflow-sync.js)

## 9. Checkpoint Discipline (v0.77.0 — non-negotiable)

Claude's context drifts as conversations get long. These checkpoints prevent that from causing mistakes.

### Mandatory Re-Read Points
Claude MUST re-read this file (`/.claude-workflow.md`) at these moments:
1. **Session start** — before doing any work
2. **Before every commit handoff** — re-read sections 4, 7, and 9 before writing the commit message
3. **After any confusion or correction from Christian** — stop, re-read, realign
4. **Before migrations** — re-read section 5 (Data Architecture) to verify key names, array structures, field names
5. **When switching phases** — coding → testing, testing → commit, commit → next task

### Verify Before Handoff
Before giving Christian a commit message, Claude checks:
- [ ] Am I queuing via OpsFlow action queue with FULL commit message in `detail` field? (NOT running git, NOT just a code block in chat)
- [ ] Did I test my migrations against actual localStorage data (not assumed data)?
- [ ] Did I update version via `OpsFlow.setVersion()` after deploy?
- [ ] Is the changelog updated?
- [ ] Is the tester updated?

### Checkpoint vs Completion (v0.91.0, refined v0.92.0)
A commit/deploy does NOT always mean tasks are done. Task groups often need multiple commits across sessions before the work is fully complete.

**Checkpoint deploy** — Getting code to production so Christian can see progress. Fast process — do the housekeeping, skip the testing:
- Version bump + changelog entry
- Update PQ statuses (set to In Progress, Deployed, create new items as needed)
- Process any new Feedback → Backlog
- Verify deploy works (quick visual check)
- Resume work immediately
- **NO** tester config, **NO** self-test cycle, **NO** lessons learned, **NO** archiving

**Task completion deploy** — The task group is fully done. Full cycle: all checkpoint steps PLUS tester config seed, self-test → testing complete → lessons learned → tasks to Done.

**The rule:** After Christian says COMMIT, Claude asks: *"Checkpoint deploy or tasks complete?"* before proceeding. Never assume completion. A task is only complete when the task description's acceptance criteria are met — not just because code was committed.

**Who archives:** Christian clicks Archive on Done items. Claude sets status to Done but NEVER moves items to Recently Completed. That's Christian's action.

### Deploy Handoff Message Format
Claude queues commit messages via the **OpsFlow action queue** on the Workflow Dashboard (workflow.html). The action banner has a "Copy Commit Msg" button that copies `action.detail` to clipboard.

**CRITICAL: `action.detail` is the FULL commit message** — title line + blank line + TYPE lines. The `action.message` is just the banner headline. The Copy button copies `detail`, NOT `message`.

**Action queue format:**
```javascript
OpsFlow.set('opsflow_action_queue', {
    id: 'action_' + Date.now(),
    type: 'commit_push',
    message: 'vX.XX.X T-### Short description',        // banner headline
    detail: 'vX.XX.X T-### Short description\n\nTYPE: Description of change\nTYPE: Description of change',  // FULL commit msg (copied by button)
    status: 'pending',
    created: new Date().toISOString()
});
```

**Commit message format** (what goes in `detail`):
```
vX.XX.X T-### Short description

TYPE: Description of change
TYPE: Description of change
```
Types: FEATURE, FIX, UPDATE, RENAME, INFRA

**Christian's steps:**
1. Click "Copy Commit Msg" on the WFD action banner
2. Paste into Visual Studio commit message window
3. Commit
4. Push
5. System runs Deploy steps in GitHub
6. Christian clicks "Done ✓" on the action banner (or tells Claude "DEPLOY COMPLETE")
7. Claude continues with Workflow

**Legacy fallback:** If the WFD action queue is unavailable, Claude provides the commit message in a code block in chat so user can copy/paste.

### When In Doubt
If Claude is unsure about any process detail: **read the doc, don't guess.** Guessing from faded context is how we get wrong migration substrings, git commits from sandbox, and JSON-quoted KV values.

## 10. Workflow Dashboard — WFD (T-119, v0.93.0+)

The Workflow Dashboard (`workflow.html`) is the live visualization of the dev cycle. It replaces text-based status updates with an interactive diagram + action queue. Admin-only page.

### Story Diagram
The WFD renders an SVG diagram showing the current task's progress as a git-branch-style flowchart:

**Trunk line** (horizontal): Start → Scan → Coding (top-level task stages, LED-colored nodes)

**Branches** fork down from the trunk and merge back up:
- **Checkpoints** (expanded): Fork down to Level 2 showing Commit Ready → Committed → Pushed → Deployed → Verify nodes. Each node is a clickable LED (red/yellow/green). Merge back to trunk after Verify.
- **Checkpoints** (collapsed): Past checkpoints collapse to a single dot below trunk with version label.
- **Discussions** (open): Fork down to a dot, no merge back (ongoing).
- **Discussions** (closed): Fork down to a dot, merge back (resolved).

**Drawing functions:**
- `drawSmoothPath(svg, waypoints, color, opts)` — Asymmetric cubic bezier (SVG `C` command). Fork DOWN: vertical departure, horizontal arrival. Merge UP: horizontal departure, vertical arrival.
- `drawRoutedPath(svg, waypoints, color, opts)` — Manhattan routing with arc corners (Q bezier). Used for connectors.
- `drawContinuationMarker(svg, x, y, num)` — Numbered circle for row-wrap connections.

**Row wrapping:** When the diagram is too wide for one row (MAX_ROW_WIDTH=1080), segments wrap to the next row. Rows are connected by continuation markers (numbered circles ①②③) — classic flowchart convention. No connector lines between rows.

**Layout constants:** `NODE_R=14, NODE_SPACING=100, LEVEL_1_Y=50, LEVEL_2_Y=150, LEFT_PAD=60, MAX_ROW_WIDTH=1080, ROW_HEIGHT=180, COLLAPSED_WIDTH=90, MINI_BRANCH_DROP=50, FORK_OFFSET=60`

**Marker spacing:** `START_MARKER_GAP=18` (left side), `END_MARKER_GAP=48` (right side, ≥2× circle diameter from last node). Trunk lines extend to be tangent to marker circles.

### Action Queue
KV key: `opsflow_action_queue`. Single object (not an array).

**Shape:** `{ id, type, message, detail, status, options, response, created, updated }`
**Types:** `commit_push`, `deploy_confirm`, `test_request`, `question`, `info`
**Status:** `pending` → `completed` | `failed` | `dismissed` | `issue`

The action banner renders at the top of the WFD when a pending action exists. Buttons: Copy Commit Msg, Report Issue, Done ✓, Dismiss.

**"Report Issue" button** creates a new discussion linked to the action. User types the issue in the discussion prompt that appears.

## 11. Key Patterns & Gotchas

- **Claude NEVER commits or pushes from the sandbox** — Claude provides the commit message in a code block. Christian commits and pushes from Visual Studio. This is non-negotiable. The sandbox creates lock files and LF line endings that break VS. Established as a work rule in v0.77.0.
- **CRLF line endings** — If VS ever shows phantom "changed" files, run `git config core.autocrlf true` in the repo, close VS, delete `.git/index.lock` and `.git/HEAD.lock`, then `git checkout -- .` to normalize.
- **Never use `-uall` flag** with git status (memory issues on large repos)
- **Read files before editing** — Edit tool requires prior Read
- **sed for simple replacements**, Read+Edit for complex changes
- **HTML entities in JS strings** — NEVER use `&amp;`, `&mdash;`, `&times;`, etc. in JavaScript strings. Use Unicode escapes (`\u2014`, `\u00D7`, `\u2022`, `&` literal) instead. HTML entities in JS strings get double-escaped by `escHtml()` or render as literal text in `innerHTML`. HTML entities in raw HTML markup (e.g., `<div>text &mdash; more</div>`) are fine. (T-088, v0.82.1)
- **Backlog HTML is ordered**: `<!-- NOW (high) -->`, `<!-- SOON (medium) -->`, `<!-- LATER (low) -->` section comments
- **Priority sort values must be non-zero** — `{ high: 1, medium: 2, low: 3 }` not `{ high: 0, ... }` because `0 || default` is falsy in JS
- **Don't clear feedback you misunderstood** — if the Backlog item description doesn't match user intent, the feedback clearing was premature
- **Self-test before handoff** — Claude should verify on live site: version lines, sort behavior, new backlog items, LED clicks
- **Read Tester results before claiming PASS** — Christian can override Claude's test results. Always re-read the Tester page after self-test before declaring final results.
- **"Rework" not "fix"** — post-deploy corrections are called reworks, not new features or fixes
- **Check time of day** when greeting Christian — don't say "good evening" at 9am
- **Don't burn credits on chat** — stay focused, avoid unnecessary back-and-forth
- **Migration keys must be unique** and should include version number
- **Migrations must be tested against real data** — read actual localStorage values before writing indexOf matches. Never assume field content.
- **Feedback clearing migrations** use `indexOf()` on `fb[i].text` to match
- **Tester fingerprint auto-resets** when TEST_SECTIONS content changes — no manual reset needed
- **PQ item IDs** start with `pq_`, **Backlog IDs** start with `bl_`, **Completed IDs** start with `done_seed_` (HTML-seeded) or keep original ID (migration-completed)
- **Migration-injected items** should use descriptive stable IDs (e.g., `backlog-delete-action`)
- **Tester completion stamp** — Claude calls `completeAITest()` after self-testing (purple "AI Self-Test" banner). Christian calls `completeTesting()` for manual tests (green "Manual Test" banner). History shows [AI] or [Manual] tags.
- **Store versions WITHOUT the "v" prefix** — `OpsFlow.setVersion('0.85.0')` not `setVersion('v0.85.0')`. The `init()` function prepends "v" for display. Storing with the prefix creates "vv0.85.0". Defensive `cleanVer` strip added in v0.85.0 rework, but correct storage is the real fix. (v0.85.0 rework)
- **Post-deploy KV version update is mandatory** — Claude must run `OpsFlow.setVersion('X.XX.X')` on the live site before self-testing. Without this, all 9 pages display the old version from stale KV data, causing a guaranteed FAIL on version checks.
- **"Back to you" means re-read the Tester** — When user says "Back to you" after testing, Claude must check for FAILs before proceeding. Don't assume PASS. Enter rework loop if FAILs exist.
- **WAIT FOR GO** — Claude must NOT start coding until Christian says GO (or "do it", "execute"). Even if the next message is clearly going to be GO, Claude waits. Discussion is cheap, premature execution is expensive and disrespectful of the user's process. If Claude catches itself about to jump the gun, ask: "Want me to GO on this?"
- **Embed workflow actions in the plan** — When building a task plan after GO, Claude must include workflow doc actions (set PQ status to In Progress, version bump, changelog entry, tester config seed, migrations) as explicit numbered steps alongside the code work. This prevents Claude from focusing on code and forgetting administrative steps. The plan IS the checklist — if it's not in the plan, it won't happen. (v0.82.1 lesson: T-088 shipped without ever being set to In Progress)
- **Test prompts before coding AI Assist** — Never code an AI integration from an untested prompt. Test by hand in 2+ LLMs first, lock the format, then build. Prompt R&D is cheap; rework after coding is not. (T-095, v0.85.0)
- **sed version bumps corrupt historical data** — Using `sed -i 's/old/new/g'` on a whole file will replace version strings inside migration keys, completedVersion fields, meta tags, and HTML comments. Always fix historical references back after a global sed. Better: target only suite-ver lines or use file-specific replacements.
- **Action queue `detail` = full commit message** — The WFD's "Copy Commit Msg" button copies `action.detail`, NOT `action.message`. The `detail` field must contain the full commit message (title line + blank line + TYPE lines). The `message` field is just the banner headline. Getting this wrong means Christian copies an incomplete message. (v0.97.2 lesson)
- **`await OpsFlow.set()` needs async IIFE** — When injecting JS via Chrome, `await` fails outside async context. Always wrap in `(async () => { await OpsFlow.set(...); })()`. (v0.96.x lesson)
- **Update workflow doc EVERY version** — After compaction, Claude falls back to this doc. If the doc is stale, Claude will use outdated processes. Every significant workflow change (new systems, format changes, new commands) must be documented here immediately, not after it causes a problem. (v0.97.2 lesson — 10 versions of drift)
- **POLL = check WFD** — When Christian types POLL, Claude refreshes the WFD, screenshots, reads new discussions, and reports status. Don't ask what POLL means.
- **`data:` and `about:blank` URLs blocked** — Chrome extension blocks navigation to `data:text/html` and `about:blank`. Use `document.write()` via JS injection instead.
